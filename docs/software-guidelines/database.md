# Database and Migrations Guidelines

## 1. Purpose

This document defines the database schema management, migration, and data seeding rules for `gam-api`.

Database migrations are a core part of the application's architecture. They must be granular, explicit, and strictly aligned with the domain model. A migration name must accurately describe the schema or data contract it introduces, and its body must contain a single, coherent responsibility.

## 2. Core Architecture Rules

### 2.1. Flyway Is the Source of Truth

Flyway exclusively manages schema creation and evolution.

* Hibernate is restricted to schema validation (`spring.jpa.hibernate.ddl-auto=validate`). It must **never** create, alter, or drop schema objects at runtime.
* Every persistent table, enum mirror, constraint, index, and database-level default must be represented by a Flyway migration.

### 2.2. Immutability of Versioned Migrations

Once a versioned migration (`V__`) is merged and applied to a shared, staging, or production database, it is strictly immutable. Future schema adjustments or corrections must be introduced via new migrations.

### 2.3. Single Coherent Responsibility

A migration must express one logical database change. Do not bundle unrelated tables or mix schema creation with demo data.

**Valid Responsibilities:**

* Creating a single table with its owned constraints and indexes.
* Creating a table along with the specific database enum it requires.
* Adding a tight, inseparable cluster of domain concepts.
* Executing a one-time data transformation.

**Forbidden Responsibilities:**

* All-in-one MVP schema dumps (e.g., creating accounts, events, and locations in one file).
* Mixing schema definitions with sample fixture data.

## 3. Schema Design and Naming

### 3.1. File Naming Convention

Migrations use descriptive names following the pattern `V<version>__<behavior>_<object>.sql`. The description must remain accurate even when read without opening the file.

* **Valid:** `V1__create_accounts_table.sql`, `V9__add_cancellation_reason_to_events.sql`.
* **Forbidden:** `V1__create_mvp_tables.sql`, `V2__update_schema.sql`.

### 3.2. Database Object Naming

Database object names must be stable, explicit, and consistent.

* **Tables:** Plural snake_case (e.g., `accounts`, `activity_logs`).
* **Enums:** Singular concept suffixed with `_enum` (e.g., `member_status_enum`).
* **Foreign Keys:** `fk_<table>_<referenced_concept>`
* **Unique Indexes:** `idx_<table>_<column_or_rule>`
* **Check Constraints:** `check_<table>_<rule>`

### 3.3. Database Enum Mirrors

Database enums act purely as mirrors of the application's domain enums. They do not define the backend contract; they follow it.

* If an enum is specific to one table, create it in the same migration file immediately before the `CREATE TABLE` statement.
* If an enum is shared across multiple domains, create it in a dedicated standalone migration.

### 3.4. Dependency Order

Migrations are ordered strictly by database dependency:

1. Standalone enum types.
2. Root tables with no foreign keys.
3. Tables with foreign keys referencing existing tables.
4. Join or assignment tables.
5. Index-only or constraint-only refinements.

## 4. Seeding and Fixture Data Strategies

Data insertion falls into three strict categories. They must never be mixed.

### 4.1. Production Seed Data (Java Repeatable Migrations)

Production seed data (e.g., system roles like `SUDO` and permissions) is required for the application to function.

* **Mechanism:** Because the source of truth for these records is often code (e.g., `PermissionEnum`), production seeding uses Java repeatable migrations (`R__SeedPermissionsAndRoles.java`).
* **Rule:** These migrations must be strictly idempotent and safe to run repeatedly across all environments.

### 4.2. Versioned Data Migrations (SQL)

Used for one-time semantic data transitions (e.g., backfilling a new non-null column or converting existing rows to a new enum value). These use standard `V__` SQL migrations.

### 4.3. Demo and Fixture Data (Isolated Callbacks)

Demo accounts, sample events, and local testing fixtures must **never** be placed in the main production migration path. They are executed via Flyway lifecycle callbacks (e.g., `afterMigrate.sql`) completely isolated from production code.

#### Directory Isolation Rules

The project enforces strict environment boundaries for migrations via `application.properties`:

**1. Production & Main Application**
Contains only production schema migrations and production seed synchronizers.

* **Location:** `src/main/resources/db/migration`
* **Config:** `spring.flyway.locations=classpath:db/migration`

**2. Local Development (Demo Data)**
Contains sample data generated by Flyway callbacks (e.g., `afterMigrate.sql`).

* **Location:** `src/main/resources/db/dev-migration`
* **Config (Dev Profile Only):** `spring.flyway.locations=classpath:db/migration,classpath:db/dev-migration`

**3. Integration Testing**
Tests default to the production schema only. Tests do **not** inherit development demo data. If a test requires seeded data, it must construct it explicitly via Java test builders or opt into dedicated test fixtures (`src/test/resources/db/test-fixtures`).
